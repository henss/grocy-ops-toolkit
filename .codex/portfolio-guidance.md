# Portfolio Guidance

This file is a self-contained shared standard for agents working directly in this repo. Keep repo-specific additions in the local block so sync can refresh the managed guidance safely.

## Coding Standard

- Prefer named exports and type-only imports.
- Keep files small, explicit, and easy to grep.
- Before non-trivial TypeScript edits, run `pnpm check:agent-surface:preedit -- <candidate files>` when available.
- Run the narrowest repo-local validation command before stopping.

## Catalog-Managed Rules

- Promote repeated workflows into reusable skills, scripts, or local automation; promote repeated mistakes or ambiguity into evals or the appropriate durable guidance surface.
- Treat useful agent judgment, recommendations, private planning, venture, creative, quality-of-life, and tool work as normal when downside is recoverable. Treat public, distribution, publishing, paid, or small ad tests as normal only when a scoped autonomy envelope records approval, cap, stop condition, measurement, and readout. Stop for dangerous, irreversible, authority-changing, life-changing, or hard-to-recover downside.
- Keep TypeScript files small and single-purpose. Put new multi-file CLI-backed features in feature-owned packages with ./cli exports and no package "." API unless a deliberate non-CLI consumer exists, keep apps/cli as composition, validate external boundaries, exclude generated API output from source-size budgets while testing the adapter seam, inject IO/time/randomness seams, keep business logic out of large UI components, place high-signal tests near code, and avoid broad barrels, ambient globals, dynamic import magic, shared mutable state, grab-bag utils, hidden build magic, and mock-heavy tests.
- Keep files small and single-purpose. Prefer explicit data shapes, boundary-local normalization, narrow helpers, and focused tests. If a touched file is already structurally busy, extract a clearer seam before adding feature weight.
- Use simple, direct language. Write for the target reader, not for the system. Prefer concrete words like issue, packet, command, file, refresh, review, approval, and blocker. Avoid vague or impressive-sounding terms such as posture, substrate, envelope, surface, chamber, doctrine, leverage, and bounded unless the text defines exactly what they mean. Keep useful detail, but remove ceremony and restatement.
- When a repo exposes Agent Atlas surfaces or rollout guidance, prefer read-only Atlas MCP tools when configured. If MCP is unavailable, run repo-advertised script wrappers such as `pnpm atlas:context-pack` and `pnpm atlas:resolve-path` before raw Atlas CLI forms; when a raw CLI form is necessary, run the subcommand help first and do not mix a positional root with `--path <root>`. Read `docs/agents/atlas.md` as the fallback orientation surface when tooling is unavailable or stale. Run `pnpm atlas:doctor` when Atlas setup looks stale or broken. If Atlas remains unavailable after a limited setup check, record the setup gap in the outcome or receipt path and continue with scoped local orientation. After non-trivial work, write an Atlas usage receipt through the repo wrapper when available, naming selected entities, files, tests, whether broad search was still needed, and missing or misleading cards; until OPS-2245 is fixed, write receipts outside `.agent-atlas` with `--out .runtime/agent-atlas/usage/<task-slug>.yaml` so validation does not parse receipts as entity cards. Do not create or expand atlas metadata speculatively; add it only when a concrete task, measured navigation waste, or cross-repo ambiguity demonstrates the need. Significant Agent Atlas bugs, gaps, or feature requests should become Linear issues through the approved Linear path instead of remaining only in chat, receipts, or session notes.
- Prefer repo-local current state, generated status, or live tool output before older notes or historical ledgers when deciding what is true now.
- When current code or state proves repo text is outdated, redundant, misleading, or wrong, repair that text in the same scoped task when safe. Prefer deleting obsolete TODOs, stale status notes, duplicate guidance, and wrong examples over adding caveats. Update the canonical source before generated copies. If cleanup crosses ownership or trust boundaries, lacks a safe render path, or rests on ambiguous evidence, record the exact stale claim and blocker in the canonical follow-up surface instead of only mentioning it in chat.
- Prefer explicit fields, frontmatter, typed records, YAML/JSON blocks, and schema-backed artifacts over parsing free-form prose to infer workflow state. If the needed signal is missing, fix the writer or add the structured field instead of extending regex or keyword heuristics.
- Prefer structural fixes over patch stacks. If repeated exceptions, guards, comments, or prompt caveats exist only to compensate for a bad abstraction or stale authority source, remove or reshape that seam instead of adding one more workaround. Keep compatibility only for known external consumers or staged migrations.
- Before broad searches, broad multi-surface planning or ideation loops, or new reusable capability work, use this repo's own capability route, catalog export, Atlas, or docs index when present. Treat generated catalogs as navigation indexes, not sources of record; verify against the registry, policy, runtime state, docs, or code they point to. If this repo has no such route, start with the narrowest local orientation surface instead of assuming an llm-orchestrator command exists.
- Record unresolved design questions, repeated confusion, token waste, compatibility ambiguity, or follow-up work in the repo's canonical tracker, decision log, evals, or outcome artifact instead of leaving it only in chat.
- Keep README as a short entrypoint. Put detailed feature, API, CLI, configuration, and architecture documentation under docs/, update the docs entrypoint when adding pages, and run repo-local docs lint when available.
- At the close of non-trivial work, note concrete efficiency waste when present and fix the root cause in the cheapest safe way, such as a clearer seam, smaller file, focused script, skill, or eval.
- Stop and surface the blocker when work crosses trust boundaries, requires real-world or external effects, changes authority assumptions, has conflicting goals, or remains low-confidence after limited retries.
- For long-running, multi-step, cross-package, or commit-per-increment implementation work, check the capability route or repo adoption overlay for `agent-workloops` before starting. Use it as the resumable work-loop state surface when adopted. When use reveals missing triggers, confusing docs, weak receipts, poor recovery, or unnecessary friction, make the cheapest safe generic improvement in the owned `agent-workloops` repo, or record a concrete follow-up there instead of leaving the gap only in chat.
- Before adding a conservative local placeholder for Reclaim, Calendar, task, shopping, household, or other external-tool work, check the owning portfolio repo for an approved adapter/CLI and bridge to it when the request is already inside delegated bounds. Self-only private task, Reclaim, and Calendar writes are delegated when the owning adapter policy and readback are healthy; gate calendar invitations, guest changes, shared-event edits, purchases, household control, and other people-facing effects. Preserve the owner repo as the execution boundary, run its health/readback path, and record the receipt.
- Before classifying work as blocked on Stefan's preference, prepare the safest decision-ready option set you can: concrete options, tradeoffs, and a recommended default. Stop before credentials, spend, public writes, account changes, or live side effects, but do not stop merely because Stefan must choose among safe options.
- When creating Reclaim tasks for deadline-sensitive work such as gifts, ordering/shipping windows, bookings, event-prep cutoffs, or purchase-candidate finalization, do not count due/snooze fields as sufficient scheduling. Use the owning repo's guard fields when present, pair the task with a private Calendar block that ends before the true completion deadline, and notify/block if no safe slot is available.
- When a coherent increment lands, commit and push it after validation passes. Stage only task-owned files unless the task intentionally overlaps already-dirty files.
- Work on this repo's current default/shared branch unless local instructions explicitly request another flow; do not create branches, worktrees, or PR-only flows for routine work.
- Treat dirty or untracked files outside the current task as active parallel work; do not delete, revert, format, normalize, or stage them unless explicitly authorized.
- Record durable decisions only for changes to architecture, authority boundaries, trust boundaries, or repo-wide operating conventions. Keep routine implementation rationale in code, tests, issues, or the local canonical artifact.
- Capture significant new understanding in the cheapest canonical memory surface: policy, registry, outcome, lesson, eval, skill, current-state artifact, decision, or approved personal knowledge-base target. Direct personal-KB writes require an approved target and readback receipt.
- For non-trivial work, state the plausible benefit path to Stefan, SMARTSEER, personal operations, creative work, finance, or another recorded objective; check the strongest likely Stefan objection; stop for protected outcomes, authority changes, public trust, money, safety, or cross-domain tradeoffs.
- Use existing repo commands, skills, Atlas surfaces, adapters, and approved tools before broad manual work. If repeated complexity, parser drift, or fragile execution appears, add or improve the thinnest reusable script, skill, eval, or adapter after respecting build-vs-buy guidance.
- For repo-internal generated artifacts primarily consumed by agents, prefer sparse structured retrieval over human-oriented narrative, keep stable headings used by tooling, and omit empty or non-material sections.
- For reusable tooling, automation, shared helpers, parser/renderers, integration glue, dependency tooling, or repeated workflow support, use Solution Scout when available; otherwise first check `registry/solutions/` plus the repo adoption overlay. Record exceptions explicitly instead of silently bypassing the shared candidate path.
- Before adding reusable tooling, shared helpers, workflow automation, connector clients, parser/renderers, supply-chain tooling, context packaging, or package-like code, use the Solution Scout MCP when available; otherwise check maintained third-party options through repo/registry/package search and record a portable scout receipt or why the work is one-off/local ownership is cheaper.
- When using repo-local skills, read the exact path surfaced in session context or `.agents/skills/<skill-name>/SKILL.md` in the current checkout; do not guess `CODEX_HOME/.codex/skills/.system` paths.
- When changing UI, remove adjacent redundant information instead of restyling it. Do not show the same concept in a section title, badge, field label, metadata row, and provenance line unless each instance supports a different decision. Optimize for scan clarity, action relevance, and lower cognitive load.
- Run the narrowest repo-local verification command that defends the change. If verification cannot pass because of unrelated dirty state, existing failures, or missing external access, record the exact command and blocker.
